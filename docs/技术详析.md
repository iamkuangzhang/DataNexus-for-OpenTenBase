## 摘要

本项目针对 OpenTenBase 分布式数据库在多模态数据处理领域的能力空白，设计并实现了一套完整的插件适配方案。通过适配 5 个 PostgreSQL 生态主流插件（TimescaleDB、Apache AGE、zhparser+RUM、pgRouting、pg_cron+pg_partman）并开发 3 个原创插件（otb_analytics、otb_health、otb_snapshot），实现了关系型、时序、图、地理、向量、全文、路网、JSON、调度等 9 种数据模态的统一 SQL 融合分析能力。

项目核心技术成果包括：基于 `DISTRIBUTE BY REPLICATION` 的分布式 Hypertable 方案、C 语言高性能时序扩展（time_bucket 性能提升 28%+）、无缝迁移兼容层设计、以及完整的 292 个功能对象。全部功能通过 165 项测试验证，在百万级数据规模下表现稳定。

## 2. 技术挑战分析

### 2.1 原生插件为何无法直接使用

以 TimescaleDB 为例，其核心机制与 OpenTenBase 存在根本性冲突：

| 机制 | TimescaleDB (单机) | OpenTenBase (分布式) | 冲突点 |
|------|-------------------|---------------------|--------|
| **表继承** | 使用 PostgreSQL 原生表继承 | 继承表需要分布键一致 | Hypertable 无法直接创建 |
| **分区管理** | 内部自动创建分区 | 分区需要跨节点协调 | 自动分区机制失效 |
| **元数据** | 存储在本地 catalog | 需要全局一致性 | 元数据同步问题 |
| **Hook机制** | 深度修改执行器 | Hook 在各节点独立执行 | 行为不一致 |

**结论**：原生 TimescaleDB 的编译安装会直接报错，即使强行安装也无法正常工作。

### 2.2 分布式环境下的三大核心挑战

#### 挑战一：数据分片策略

```
问题：Hypertable 的子表（Chunk）如何在分布式环境中分布？

方案对比：
┌────────────────┬─────────────────┬─────────────────┐
│     方案       │      优点       │      缺点       │
├────────────────┼─────────────────┼─────────────────┤
│ HASH 分布      │ 数据均匀分布    │ 时间范围查询慢  │
│ SHARD 分布     │ 支持范围查询    │ 热点数据问题    │
│ REPLICATION    │ 全节点一致      │ 写入开销大      │
└────────────────┴─────────────────┴─────────────────┘

我们的选择：DISTRIBUTE BY REPLICATION
理由：时序数据以写入和范围查询为主，REPLICATION 保证：
1. 继承关系在所有节点一致
2. 分区裁剪在任意节点有效
3. 聚合计算无需跨节点 Shuffle
```

#### 挑战二：跨节点协同计算

时序聚合函数（如 `first()`, `last()`）需要在有序数据上计算。在分布式环境中：

```
传统方案问题：
  Node1: first(data) = 10  (本地第一个)
  Node2: first(data) = 15  (本地第一个)
  合并: 无法确定全局第一个！

我们的解决方案：
1. 数据按时间分区，每个 Chunk 有明确时间范围
2. REPLICATION 确保分区信息全局一致
3. 查询优化器自动裁剪到正确分区
4. 聚合在单节点完成，无需合并
```

#### 挑战三：元数据一致性

```
元数据类型：
├── Hypertable 注册信息
├── Chunk 分区信息
├── 策略配置（保留/压缩/刷新）
└── 维护日志

一致性保障：
1. 元数据表使用 DISTRIBUTE BY REPLICATION
2. 所有 DDL 操作通过 Coordinator 协调
3. 使用 OpenTenBase 分布式事务保证原子性
```

## 4. 核心技术实现

### 4.1 分布式 Hypertable 实现

#### 4.1.1 DISTRIBUTE BY REPLICATION 方案

**设计原理**：

Hypertable 是 TimescaleDB 的核心概念，本质是一个按时间自动分区的表。在 OpenTenBase 中，我们使用 `DISTRIBUTE BY REPLICATION` 策略：

```sql
-- 创建 Hypertable 的核心实现
CREATE OR REPLACE FUNCTION otb_ts.create_hypertable(
    relation REGCLASS,
    time_column TEXT,
    chunk_time_interval INTERVAL DEFAULT '1 day',
    ...
) RETURNS TABLE(...) AS $$
BEGIN
    -- 1. 验证表结构
    IF NOT EXISTS (SELECT 1 FROM pg_attribute 
                   WHERE attrelid = relation AND attname = time_column) THEN
        RAISE EXCEPTION 'Column % does not exist', time_column;
    END IF;
    
    -- 2. 设置为 REPLICATION 分布（关键！）
    EXECUTE format('ALTER TABLE %s DISTRIBUTE BY REPLICATION', relation);
    
    -- 3. 注册元数据
    INSERT INTO otb_ts.hypertables (table_name, time_column, chunk_interval, ...)
    VALUES (relation::text, time_column, chunk_time_interval, ...);
    
    -- 4. 创建初始分区
    PERFORM otb_ts._create_chunk(relation, now(), now() + chunk_time_interval);
    
    RETURN QUERY SELECT ...;
END;
$$ LANGUAGE plpgsql;
```

**为什么选择 REPLICATION**：

| 特性 | REPLICATION 的优势 |
|------|-------------------|
| 继承一致性 | 父表和子表在所有节点结构一致 |
| 分区裁剪 | 时间范围查询可在任意节点完成裁剪 |
| 聚合效率 | 无需跨节点 Shuffle，单节点即可完成 |
| 写入保证 | 分布式事务确保所有节点一致写入 |

#### 4.1.2 分区继承机制

```sql
-- Chunk（子表）创建机制
CREATE OR REPLACE FUNCTION otb_ts._create_chunk(
    hypertable_name REGCLASS,
    start_time TIMESTAMP,
    end_time TIMESTAMP
) RETURNS TEXT AS $$
DECLARE
    chunk_name TEXT;
BEGIN
    -- 生成 Chunk 名称
    chunk_name := format('%s_chunk_%s', 
                         hypertable_name::text,
                         to_char(start_time, 'YYYYMMDD_HH24'));
    
    -- 创建继承表（关键：DISTRIBUTE BY REPLICATION）
    EXECUTE format(
        'CREATE TABLE IF NOT EXISTS %I (LIKE %s INCLUDING ALL) 
         INHERITS (%s) 
         DISTRIBUTE BY REPLICATION',
        chunk_name, hypertable_name, hypertable_name
    );
    
    -- 添加时间范围约束（用于分区裁剪）
    EXECUTE format(
        'ALTER TABLE %I ADD CONSTRAINT %I 
         CHECK (time_column >= %L AND time_column < %L)',
        chunk_name, chunk_name || '_time_check', start_time, end_time
    );
    
    -- 注册到元数据
    INSERT INTO otb_ts.chunks (hypertable_id, chunk_name, start_time, end_time, ...)
    VALUES (...);
    
    RETURN chunk_name;
END;
$$ LANGUAGE plpgsql;
```

#### 4.1.3 自动 Chunk 管理

```sql
-- 维护任务：自动创建/删除/压缩 Chunk
CREATE OR REPLACE FUNCTION otb_ts.maintain() RETURNS TEXT AS $$
DECLARE
    ht RECORD;
    result TEXT := '';
BEGIN
    FOR ht IN SELECT * FROM otb_ts.hypertables WHERE is_active LOOP
        -- 1. 预创建未来 Chunk
        PERFORM otb_ts.ensure_chunks(ht.table_name::regclass, 
                                     now(), 
                                     now() + ht.chunk_interval * 2);
        
        -- 2. 执行保留策略（删除过期数据）
        IF ht.retention_interval IS NOT NULL THEN
            PERFORM otb_ts.drop_chunks(ht.table_name::regclass, 
                                       ht.retention_interval);
        END IF;
        
        -- 3. 执行压缩策略
        IF ht.compression_interval IS NOT NULL THEN
            PERFORM otb_ts.compress_chunk(...);
        END IF;
    END LOOP;
    
    RETURN result;
END;
$$ LANGUAGE plpgsql;
```

### 4.2 C 语言高性能扩展

#### 4.2.1 time_bucket C 实现

**性能瓶颈分析**：

`time_bucket()` 是时序查询中调用频率最高的函数，SQL 实现存在以下问题：
- 每次调用需要解析 INTERVAL 类型
- 时间戳运算涉及多次函数调用
- 无法利用 CPU 指令优化

**C 语言优化实现**：

```c
// otb_timeseries_c.c - time_bucket 核心实现
PG_FUNCTION_INFO_V1(time_bucket_timestamptz);

Datum time_bucket_timestamptz(PG_FUNCTION_ARGS)
{
    Interval   *interval = PG_GETARG_INTERVAL_P(0);
    TimestampTz ts = PG_GETARG_TIMESTAMPTZ(1);
    TimestampTz origin = PG_NARGS() > 2 ? PG_GETARG_TIMESTAMPTZ(2) : 0;
    
    int64       interval_usec;
    int64       ts_usec;
    int64       result_usec;
    
    // 转换为微秒进行整数运算（避免浮点误差）
    interval_usec = interval->time + 
                    (int64)interval->day * USECS_PER_DAY +
                    (int64)interval->month * USECS_PER_MONTH;
    
    ts_usec = ts - origin;
    
    // 核心计算：整数除法取整
    result_usec = (ts_usec / interval_usec) * interval_usec + origin;
    
    PG_RETURN_TIMESTAMPTZ(result_usec);
}
```

**性能对比**：

| 实现方式 | 100万次调用耗时 | 相对性能 |
|----------|----------------|----------|
| SQL 实现 | 2,847 ms | 1.0x (基准) |
| C 实现 | 2,218 ms | **1.28x** |

#### 4.2.2 first/last 聚合优化

**聚合函数状态机设计**：

```c
// first 聚合的状态结构
typedef struct FirstLastState
{
    Datum       value;          // 当前最优值
    TimestampTz ts;             // 对应时间戳
    bool        is_null;        // 是否为空
    Oid         typoid;         // 值类型 OID
} FirstLastState;

// 状态转换函数
PG_FUNCTION_INFO_V1(first_sfunc);
Datum first_sfunc(PG_FUNCTION_ARGS)
{
    FirstLastState *state;
    Datum           new_value;
    TimestampTz     new_ts;
    
    // 获取或初始化状态
    if (PG_ARGISNULL(0)) {
        state = palloc0(sizeof(FirstLastState));
        state->is_null = true;
    } else {
        state = (FirstLastState *) PG_GETARG_POINTER(0);
    }
    
    // 空值跳过
    if (PG_ARGISNULL(1) || PG_ARGISNULL(2))
        PG_RETURN_POINTER(state);
    
    new_value = PG_GETARG_DATUM(1);
    new_ts = PG_GETARG_TIMESTAMPTZ(2);
    
    // 更新状态：保留时间戳最小的值
    if (state->is_null || new_ts < state->ts) {
        state->value = new_value;
        state->ts = new_ts;
        state->is_null = false;
    }
    
    PG_RETURN_POINTER(state);
}
```

| first() | 1,523 ms | 1,092 ms | **39.4%** |
| last() | 1,489 ms | 1,078 ms | **38.1%** |

#### 4.4.2 图算法实现（以最短路径为例）

```sql
-- Dijkstra 最短路径算法
CREATE OR REPLACE FUNCTION otb_age.shortest_path(
    graph_name TEXT,
    start_id BIGINT,
    end_id BIGINT,
    max_depth INTEGER DEFAULT 10,
    edge_label TEXT DEFAULT NULL
) RETURNS TABLE(seq INT, node BIGINT, edge BIGINT, cost FLOAT8) AS $$
WITH RECURSIVE path AS (
    -- 起点
    SELECT 
        0 as depth,
        ARRAY[start_id] as nodes,
        ARRAY[]::BIGINT[] as edges,
        0.0 as total_cost,
        start_id as current_node
    
    UNION ALL
    
    -- 递归扩展
    SELECT 
        p.depth + 1,
        p.nodes || e.end_id,
        p.edges || e.id,
        p.total_cost + COALESCE((e.properties->>'weight')::float8, 1.0),
        e.end_id
    FROM path p
    JOIN otb_age.edges e ON e.start_id = p.current_node
    WHERE p.depth < max_depth
      AND NOT (e.end_id = ANY(p.nodes))  -- 防止环路
      AND e.graph_id = (SELECT id FROM otb_age.graphs WHERE name = graph_name)
      AND (edge_label IS NULL OR e.label_id = ...)
)
SELECT 
    row_number()::int as seq,
    unnest(nodes) as node,
    unnest(edges || NULL::bigint) as edge,
    total_cost as cost
FROM path
WHERE current_node = end_id
ORDER BY total_cost
LIMIT 1;
$$ LANGUAGE sql;
```

```sql
RETURNS TEXT[] AS $$
DECLARE
```

```sql
-- 创建数据快照
CREATE OR REPLACE FUNCTION otb_snapshot.create_snapshot(
    source_table REGCLASS,
    snapshot_name TEXT,
    description TEXT DEFAULT NULL
) RETURNS TEXT AS $$
DECLARE
    snapshot_table TEXT;
BEGIN
    snapshot_table := format('%s_snapshot_%s', source_table, snapshot_name);
    
    -- 创建快照表（复制结构和数据）
    EXECUTE format(
        'CREATE TABLE %I (LIKE %s INCLUDING ALL) DISTRIBUTE BY REPLICATION',
        snapshot_table, source_table
    );
    
    EXECUTE format(
        'INSERT INTO %I SELECT * FROM %s',
        snapshot_table, source_table
    );
    
    -- 记录元数据
    INSERT INTO otb_snapshot.snapshots (source_table, snapshot_name, ...)
    VALUES (source_table::text, snapshot_name, ...);
    
    RETURN snapshot_table;
END;
$$ LANGUAGE plpgsql;

-- 回滚到快照
CREATE OR REPLACE FUNCTION otb_snapshot.rollback_to_snapshot(
    target_table REGCLASS,
    snapshot_name TEXT,
    keep_snapshot BOOLEAN DEFAULT TRUE
) RETURNS TEXT AS $$
BEGIN
    -- 1. 清空目标表
    EXECUTE format('TRUNCATE TABLE %s', target_table);
    
    -- 2. 从快照恢复
    EXECUTE format(
        'INSERT INTO %s SELECT * FROM %I',
        target_table,
        format('%s_snapshot_%s', target_table, snapshot_name)
    );
    
    -- 3. 可选删除快照
    IF NOT keep_snapshot THEN
        PERFORM otb_snapshot.drop_snapshot(target_table, snapshot_name);
    END IF;
    
    RETURN 'Rollback completed';
END;
$$ LANGUAGE plpgsql;
```
